# План расширения: SEO/технический аудит на проверкасайтов.рф

> **Статус:** главный планирующий документ второго сервиса (кодовое имя внутри lawcheck: **Сервис Б / seo**).
> **Дата:** 2026-07-15.
> **Опирается на:** [site-audit-catalog.md](site-audit-catalog.md) (каталог проверок, приоритеты P0–P3, free/paid-модель), [site-audit-oss-stack.md](site-audit-oss-stack.md) (green-light OSS-стек), вердикт архитектурной панели (subdirectory, 8.7), карту семантики двух ядер, разведку rechecker.ru, инвентаризацию кода.
> **Терминология:** Сервис А = юридическая проверка (текущий продукт, 21 чек), Сервис Б = SEO/технический аудит (новый).

---

## 1. Резюме решения

Сервис Б запускается **подкаталогом `/seo/` на основном домене проверкасайтов.рф** — вердикт архитектурной панели (subdirectory 8.7 против hybrid 7.9 и subdomain 4.8). Решающий аргумент — Яндекс: поддомен для него отдельный сайт с нулевыми хостовыми факторами и риском склейки в зеркало, а подкаталог наследует возраст, цитируемость и поведенческие домена с первого дня; для кириллического IDN поддомен дополнительно означает punycode-хост в DNS/сертификатах и сужение выбора CA при нулевом SEO-выигрыше. Все референсы ниши (pr-cy, be1, rechecker, sitechecker) собирают органику подкаталогами. Технически это route-сегмент в существующем Next.js-репозитории с общими BullMQ/Postgres/SSE — минимальная цена для соло-разработчика. Сервисы продают друг друга контекстными кросс-блоками: пограничные инструменты Б (SSL, security-заголовки, WHOIS) ведут на бесплатный юр-скан А («Технически сайт в порядке? Проверьте по 21 закону РФ»), а юр-отчёты А (бесплатный и за 700 ₽) — на технический скан того же URL в один клик через общую БД сканов. Раздел `/seo/` проектируется как самодостаточный переносимый силос (стабильные латинские slug'и, свой breadcrumb-корень, своя секция sitemap) — страховка на случай гипотетического отделения в отдельный латинский домен одной картой 301.

## 2. Архитектура и позиционирование

### 2.1. Сравнение вариантов (счёты судьи)

| Вариант | Счёт | Ключевой аргумент |
|---|---|---|
| **Подкаталог `/seo/`** — победитель | **8.7** | Наследует хостовые факторы домена в Яндексе с первого дня; консенсус всех 4 референсов ниши; один репозиторий/деплой/сертификат/пара Вебмастер+GSC. Каннибализация гасится редакционными правилами (§4) |
| Гибрид (подкаталог + миграционная готовность) | 7.9 | Этап 1 идентичен подкаталогу; «зонтичный слой» на главной отвергнут как нарушающий требование владельца №1. Всё ценное (переносимость силоса) включено в условия победителя |
| Поддомен | 4.8 | 6–12+ мес нулевой кармы в Яндексе, риск склейки в зеркало, punycode-хост `seo.xn--80a…`, двойная операционка навсегда |

### 2.2. Обязательные условия реализации (из вердикта)

1. Хаб `/seo/` + каждый инструмент на отдельном URL `/seo/<slug>/` со своим Title/H1. Никогда «одна страница на все проверки».
2. Один маркерный кластер — один URL. Широкие «проверка сайта», «проверить сайт онлайн», «SEO-аудит сайта» таргетирует **только** хаб `/seo/`; «аудит сайта» и «технический аудит сайта» — **только** посадочная платного полного аудита `/seo/full-audit/` (§3.1); «анализ сайта» закрывается вхождением в description/подзаголовок хаба, без отдельной страницы. Одна страница с одним Title не может быть релевантной 4–5 маркерным запросам (референс: pr-cy держит `/analysis/` и `/audit/` отдельными URL).
3. Главную А не трогать: максимум один пункт навигации + один кросс-блок.
4. Никаких тонких страниц: инструмент публикуется только с формой в hero + 1500–2000 знаков текста + FAQ 4–5 вопросов + крошки + «похожие проверки»; FAQPage + BreadcrumbList schema с первого дня. Запуск волнами по готовности контента.
5. `/seo/` — переносимый силос: свой breadcrumb-корень, отдельная секция sitemap, стабильные slug'и, зафиксированный URL-контракт.
6. Никаких нетематических утилит (CSS-генераторы, QR, пароли) — только проверки сайта. Это доказанная слабость rechecker (из 141 инструмента лишь 24 SEO + 12 security, остальное размывает тематичность).
7. Кросс-селл только контекстными блоками в теле страниц (§5).
8. Поддомен допустим позже только для кабинета/приложения с латинским лейблом (`app.` / `lk.`); wildcard-сертификаты для IDN не использовать — per-host Let's Encrypt в punycode.
9. Гигиена семантики: перекластеризация и проверка пересечений ядер каждые 6–9 мес; canonical/301 — только к реальным дублям.
10. Аналитика: сегменты Метрики и GSC по каталогам (отдельные цели для `/seo/`), без второго property.

### 3.3. Sitemap / robots

- **Sitemap:** **зафиксировано** — sitemap-index с отдельным `/sitemap-seo.xml` (единственный вариант, дающий раздельную статистику индексации по силосам в Вебмастере и GSC): только хаб, инструменты, `/seo/full-audit/`, посадочные мониторинга, pricing. Отчёты `/seo/report/*` в sitemap не попадают.
- **robots:** `Disallow:` — **только** `/api/`. Отчёты `/seo/report/*` (и `/report/*` сервиса А) исключаются исключительно через meta robots `noindex` / X-Robots-Tag, **без** Disallow: при Disallow робот не скачает страницу, никогда не увидит noindex, и расшаренные пользователями отчёты попадут в индекс как «Проиндексировано, несмотря на блокировку в robots.txt» (URL без контента). Если `/report/` сервиса А сейчас закрыт через Disallow — убрать в Фазе 0.
- **Canonical:** каждый инструмент — self-referencing; никаких canonical между тематически смежными страницами А и Б (условие 9).
- **Метрика/GSC:** сегмент «страница начинается с /seo/», отдельные цели «запуск SEO-скана», «переход Б→А», «переход А→Б».

### 3.4. Навигация внутри силоса Б

Общий header несёт nav сервиса А («Как работает / Тарифы / Проверить сайт») — на страницах `/seo/*` эти пункты вели бы в юр-воронку, путали посетителя и нарушали правило §4.4. Layout-силос `/seo/` получает **свой вариант header**: «Инструменты» (каталог на хабе) / «Тарифы» → `/seo/pricing/` / «Экспресс-аудит» (форма хаба) + один обратный пункт «Проверка по законам РФ» → `/`. Симметрично на страницах А остаётся один пункт «SEO-аудит» → `/seo/`.

## 4. Правила анти-каннибализации

**Распределение запросов** (из карты семантики — выдача по широким запросам на 80–90 % технико-SEOшная, юр-интента в топе нет):

|---|---|---|
| «проверка сайта», «проверить сайт онлайн», «SEO-аудит сайта» | **только Б** | `/seo/` |
| «аудит сайта», «технический аудит сайта», «аудит сайта цена» | **только Б** | `/seo/full-audit/` |
| «анализ сайта» | Б | вхождение в description/подзаголовок `/seo/`, отдельной страницы нет |
| «мониторинг сайта», «мониторинг доступности/uptime», «мониторинг SSL-сертификата» | Б | `/seo/uptime-monitor/`, `/seo/ssl-monitor/`, `/seo/seo-monitor/` (не pricing) |
| «скорость», «SSL», «robots.txt», «битые ссылки», «мета-теги» и прочие узкие инструментальные | Б | `/seo/<slug>/` |
| «проверка сайта на 152-ФЗ / по законам РФ», «юридический аудит сайта», «правовой аудит» | А | `/` и страницы А |
| «cookie-баннер 152-ФЗ», «РКН», «политика конфиденциальности», «штрафы ПДн», «реестр операторов» | А | волна А-1 (§6): `/cookie-banner/`, `/rkn-registry/`, статьи о штрафах |
| «проверить сайт на мошенничество» | никто | — |

Принцип таблицы: **один маркерный кластер — один URL**; два URL, претендующих на один маркерный запрос, — дефект, который правится Title/контентом (правило 6).

**Жёсткие правила:**

1. Каждый Title страниц А содержит юр-квалификатор («152-ФЗ», «по законам РФ», «РКН», «юридический», «штраф»). В Title/H1/description страниц Б эти слова запрещены.
2. Голое «аудит сайта» принадлежит Б; А употребляет только «юридический аудит сайта» / «аудит сайта по 152-ФЗ».
3. Слово «РКН» без уточнения не использовать в Title даже у А — уезжает в чужой интент проверки блокировок.
4. **Перелинковка:** А↔Б ссылаются только кросс-блоками из §5 и одним пунктом nav; никаких сквозных ссылочных подвалов «все инструменты» на страницах А и наоборот. Внутри Б — плотная перелинковка хаб↔инструменты↔«похожие проверки» (силос).
5. **Canonical/301** — только для реальных дублей внутри одного силоса; между А и Б никогда.
6. Контроль: каждые 6–9 месяцев перекластеризация ядер и проверка пересечений (site:/inurl:, Arsenkin); если страница А и Б начали конкурировать за один запрос — правится Title/контент, не canonical.
7. Чек-лист анти-каннибализации оформляется **артефактом** `docs/seo-dev/page-checklist.md` (заводится в Фазе 0): Title без юр-слов и без брендового суффикса, один маркерный кластер на URL, крошки от `/seo/`, self-canonical, 1500–2000 знаков, FAQPage/BreadcrumbList schema, description по паттерну §2.3. Прохождение чек-листа каждой новой страницей — обязательный пункт PR-review и DoD Фаз 1, 2 и 4.

## 5. Кросс-продажи

Принцип: только контекстные блоки в теле страниц/отчётов, ни одного второго hero или зонтичного оффера. Общая инфраструктура (одна таблица `scans` с дискриминатором, единый будущий аккаунт) даёт «повторный скан того же URL другим сервисом» бесплатно.

| Где | Блок | Ведёт |
|---|---|---|
| Юр-отчёт А (бесплатный и за 700 ₽), после списка нарушений | «С законом разобрались — а как со скоростью и SEO? Технический аудит этого сайта — бесплатно» + кнопка с предзаполненным URL | Прямой POST в API Б с тем же URL → `/seo/report/[id]` (допустимо: форма Б согласий не требует) |
| SEO-отчёт Б | «Технически сайт в порядке? Проверьте его по 21 закону РФ — бесплатно» + кнопка с предзаполненным URL | Форма А с предзаполненным URL: ScanForm А содержит два **обязательных** чекбокса согласий (ПДн/оферта), поэтому прямой POST в API А «в один клик» недопустим — юр-нестыковка для сервиса, продающего соответствие 152-ФЗ. Альтернатива: кросс-блок с теми же чекбоксами прямо в отчёте Б |
| Пограничные инструменты Б: `/seo/ssl/`, `/seo/security-headers/`, `/seo/whois/`, `/seo/dns/` | В теле страницы (не hero): «SSL есть, а 152-ФЗ соблюдён? Безопасность — это ещё и закон о персональных данных» | `/` |
| Главная А | Один кросс-блок перед футером (см. §3.1) | `/seo/` |
| Хаб `/seo/` | Один сдержанный блок «Нужна проверка по законам РФ?» | `/` |
| Header | На страницах А — пункт nav «SEO-аудит» → `/seo/`; на страницах Б — свой header силоса (§3.4) с обратным пунктом «Проверка по законам РФ» | `/seo/` ↔ `/` |

**Чего не делаем:** общей «мега-формы» с выбором типа проверки на главной (размывает конверсию А — требование владельца №1); бандл-тариф «юр + SEO» отложен до появления аккаунтов и статистики кросс-перетока (открытый вопрос §7). Модель free→paid у Б повторяет уже работающую у А: «1 страница бесплатно → весь сайт / мониторинг платно».

## 6. План выполнения

Привязка к коду — из инвентаризации: работает путь `POST /api/scan → BullMQ scan → worker.ts → SSE → /report/[id]`; чеки задуманы как чистые функции от ScanContext (registry-паттерн ПС-03); UI-примитивы (GlassCard, Button, Input, ScoreRing, header/footer) generic.

### Фаза 0 — фундамент (аддитивно, без переписывания)

- **БД (drizzle-kit generate):** колонка `scanType` pgEnum `['law','seo']` default `'law'` в `scans`. Результаты Б — в существующую `check_results` (`checkId` — свободный text, `evidence` — JSONB); SEO-специфику класть в `evidence.extra`, enum `severity` не трогать.
- **Очередь:** `auditQueue` (name `audit`) в `src/server/queue/index.ts` тем же singleton-паттерном; concurrency 1–2 (headless Chrome). Отдельный процесс `worker-seo.ts` (скрипт `worker:seo:dev`) — app по-прежнему не запускает браузер.
- **Стадии:** обобщить `ScanStage` в `src/lib/constants.ts` (SEO-стадии: `fetch|lighthouse|checks|scoring` или `stage: string`), `STAGE_LABEL`/пороги вердикта — в конфиг по `scanType`. `publish.ts`/`sse.ts`/`redis.ts` и SSE-роут не трогаем (id-based).
- **API:** `POST /api/audit` (Zod strict, normalizeUrl, дедуп, envelope — всё переиспользуется). `GET /api/scans/[id]` и SSE обслуживают оба типа после обобщения сериализации.
- **next.config.ts:** `lighthouse`, `playwright`/`puppeteer` в `serverExternalPackages`.
- **DoD:** миграция применена; `POST /api/audit` ставит job в `audit`; `worker-seo.ts` публикует стадии в SSE; юр-путь регрессионно не сломан (`GET /api/health` зелёный, существующий скан проходит); проведён аудит текущих Title/H1/description **всех** страниц А на соответствие §4.1 — Title главной содержит «по законам РФ» или «152-ФЗ» (если нет — правится, иначе главная А сама конкурирует с хабом Б за голую «проверку сайта»); `/report/` убран из `Disallow` в robots.txt в пользу noindex (§3.3); заведён артефакт `docs/seo-dev/page-checklist.md` (§4.7).

### Фаза 1 — MVP: экспресс-аудит (P0) + лендинг + волна 1а

- **Движок:** `src/server/audit/**` по образцу ПС-03 — свой `AuditContext` (HTML через got/crawlee + cheerio, `ssl-checker`, lighthouse одной страницы) и чистые чек-функции в registry: мета-теги (title/description/H1/OG/Twitter/canonical), robots/sitemap наличие, редиректы, SSL, security-заголовки (правила http-observatory + `csp_evaluator`), CWV из Lighthouse. Плюс дешёвые чеки на уже загруженном HTML, чтобы флагман честно давал заявленные хабом «25+ проверок» (иначе убрать цифру с хаба до Фазы 2): alt-тексты изображений (cheerio), mixed-content, наличие JSON-LD/Schema.org, viewport/charset/favicon, CMS-детект (`wappalyzergo`). Скоринг — своя формула penalty (цвета/пороги ScoreRing переиспользуются).
- **OSS:** `cheerio`, `metascraper`, `robots-parser`, `sitemapper`, `got`, `ssl-checker`, `csp_evaluator`, `lighthouse`.
- **Фронт:** layout-силос `src/app/seo/` (свой breadcrumb-корень, свой header по §3.4); хаб `/seo/` с формой (упрощённая ScanForm без двух чекбоксов согласия); отчёт `/seo/report/[id]` — параметризованный ReportView (лейблы стадий, вердикт по типу); шаблонный компонент tool-страницы (форма+текст+FAQ+крошки+похожие) с FAQPage/BreadcrumbList JSON-LD.
- **Защита формы:** SSRF-guard (блок приватных IP, DNS-rebinding, редиректов на localhost), rate-limit и captcha реализуются **здесь, в Фазе 1 Б**, и переиспользуются сервисом А — а не наоборот: зависимость от сроков «Фазы 2 юр-плана» снята, публичный запуск Б без этой защиты невозможен (риск 4).
- **Контент-волна 1а (только с готовыми текстами):** `/seo/meta-tags/`, `/seo/speed/`, `/seo/ssl/`, `/seo/security-headers/` + кросс-блоки в оба отчёта.
- **DoD:** экспресс-аудит любого URL проходит end-to-end за ≤60 с; `POST /api/audit` защищён SSRF-guard, rate-limit и captcha включены; 4 tool-страницы опубликованы с полным контент-обвесом и прошли чек-лист `page-checklist.md`; schema валидна в валидаторе; sitemap-секция и robots-правила задеплоены; сегмент Метрики `/seo/` создан.

### Фаза А-1 — минимальная волна обвязки сервиса А (параллельно Фазам 1–2)

Свободные ниши ядра А не должны простаивать, пока ресурсы уходят в Б, — их могут занять конкуренты (pdaudit, fzscan):

- `/cookie-banner/` — чекер cookie-баннера (свободная ниша по карте семантики);
- `/rkn-registry/` — проверка в реестре операторов РКН по ИНН (соответствующий чек уже есть среди 21);
- 1–2 статьи о штрафах ПДн (лид-магнит; конструктор политики конфиденциальности — кандидат следующей волны А).

Приоритет: старт после DoD Фазы 1 Б, завершение до начала Фазы 3. Контент-обвес и чек-лист — те же требования, что у страниц Б (§4); Title каждой страницы содержит юр-квалификатор (§4.1).

- **DoD:** обе tool-страницы А опубликованы с полным контент-обвесом; опубликована минимум одна статья о штрафах; все страницы прошли `page-checklist.md`.

### Фаза 2 — паритет P1: краулинговые инструменты + платный полный аудит + лайт-мониторинг

- Инструменты волны 1б: `/seo/robots-txt/`, `/seo/sitemap/`, `/seo/redirects/`, `/seo/broken-links/`, `/seo/hreflang/` (`linkinator`, `sitemapper`, `robots-parser`, `got`).
- **Платный полный аудит** (модель rechecker «1 страница free → весь сайт 49–499 ₽»): `unlighthouse`/`crawlee` в `worker-seo.ts`, лимит страниц по тарифу, оплата через существующую payments/deliver-инфраструктуру А (YooKassa, PDF через puppeteer — deliver-очередь уже заведена). Посадочная — `/seo/full-audit/` (§3.1), владелец кластеров «аудит сайта» / «технический аудит сайта» / «аудит сайта цена».
- **Разовые платные батчи** (§6.3 каталога — третья опора модели «free-крючок → мониторинг → батчи»): sitemap до 25k URL, битые ссылки/изображения до сотен страниц, 99–499 ₽ — переиспользуют те же `linkinator`/`sitemapper` и payments этой фазы, лимит страниц по оплате. Батч HTML-валидации (до 100 стр.) — после появления инструмента в Фазе 4.
- **Лайт-мониторинг без кабинета** — осознанная правка фазировки: roadmap §6.5 каталога ставит «Uptime + SSL Monitor + Telegram» в P0 («ядро recurring-выручки, технически простое»), но полный мониторинг требует аккаунтов и биллинга подписок (Фаза 3). Разрыв закрывается здесь: Uptime+SSL monitor с управлением по magic-link на email — P0 recurring работает до появления кабинета, в Фазе 3б мигрирует в кабинет.
- **DoD:** все P1-инструменты в проде и прошли чек-лист; полный аудит на N страниц оплачивается и доставляется PDF; `/seo/full-audit/` и `/seo/pricing/` опубликованы; минимум один батч-режим продаётся; лайт-монитор Uptime/SSL создаётся по email и шлёт алерт.

Прежняя монолитная «Фаза 3» (аккаунт + auth + кабинет + история + биллинг подписок + 4 монитора + schedulers + heartbeat + алерты + weekly-отчёты) — 2–3 месяца работы соло-разработчика без промежуточных релизов; разбита на две под-фазы со своими DoD.

### Фаза 3а — единый аккаунт, история сканов, биллинг подписок

- Единый аккаунт (email + magic-link), кабинет, история сканов **обоих типов** (law+seo) по URL пользователя (кросс-селл §5). Закрывает бывший открытый вопрос 3: аккаунт вводится до мониторинга.
- **Биллинг подписок** — отдельный контур, не покрытый разовыми платежами А за 700 ₽: рекуррентные списания YooKassa (сохранённый способ оплаты, автопродление, отмена) **или** предоплатный баланс — выбор модели по открытому вопросу 1 (unit-экономика микроцен). Проектируется здесь, потребляется Фазой 3б.
- **DoD:** пользователь регистрируется, видит историю сканов обоих типов (law+seo) по своим URL; способ оплаты сохраняется, тестовое рекуррентное списание и отмена подписки проходят.

### Фаза 3б — мониторинг (recurring-ядро монетизации)

- Мониторы по каталогу §6.2, в порядке простоты: сначала **Uptime (50 ₽/мес) и SSL (30 ₽/мес)** — миграция лайт-мониторов Фазы 2 в кабинет; затем отдельным инкрементом SEO-monitor diff Title/Description/H1/canonical (20 ₽/мес) и еженедельный отчёт (99 ₽/мес); бандл ~199 ₽/мес.
- Индексируемые посадочные мониторинга из §3.1: `/seo/uptime-monitor/`, `/seo/ssl-monitor/`, `/seo/seo-monitor/`.
- **Архитектура:** BullMQ **Job Schedulers** (`upsertJobScheduler`, детерминированный `schedulerId = monitor:${siteId}`), раздельные очереди `checks` (быстрые, высокий concurrency) / `audits` (тяжёлые) / `reports` (weekly, с `tz`); концепции uptime-kuma (heartbeat-таблица, uptime %, degradation). Новые таблицы `monitors`, `heartbeats` (здесь — единственное место, где нужны отдельные seo_*-таблицы). Алерты: Telegram + email (SMTP уже в env).
- **DoD:** монитор создаётся/оплачивается/шлёт алерт в Telegram; пользователь управляет мониторами из кабинета; weekly-отчёт приходит по понедельникам; смена тарифа идемпотентно апдейтит scheduler; при неуплате монитор ставится на паузу, при оплате возобновляется идемпотентно; посадочные мониторинга опубликованы и прошли чек-лист.

### Фаза 4 — P2: закрытие gap + дифференциация

- `/seo/schema/` (`schema-dts` + `jsonld.js`), `/seo/html-validator/` (`html-validate`), `/seo/dns/` + `/seo/whois/` (`node:dns`, `whoiser`; для `.ru`/`.рф` проверить RDAP), `/seo/a11y/` (`axe-core` или `equal-access`) — a11y почти нет у конкурентов, окно дифференциации.
- `/seo/serp-preview/` (SERP/OG Preview) и `/seo/email-deliverability/` (SPF/DKIM/DMARC, `mailauth`) — слабоконкурентные кластеры волны 2, стек готов.
- Батч-режим HTML-валидации до 100 стр. (§6.3 каталога) поверх `/seo/html-validator/` — на payments Фазы 2.
- Второй козырь дифференциации: CrUX field-данные (`crux-api`) в отчёте скорости + история/тренды метрик в кабинете (`lighthouse-ci`-схема хранения, tremor-дашборды).
- **DoD:** P2-инструменты в проде и прошли чек-лист; в отчёте скорости lab+field; график деградации в кабинете.

### Фаза 5 — бэклог (скоуп и DoD — отдельным решением после метрик Фазы 3)

Статус — именно **бэклог**, не фаза с обязательствами: скоуп и DoD фиксируются отдельным решением по итогам метрик Фаз 3а/3б. Кандидаты: bulk-проверка URL, REST API pay-per-request, AI-инструменты, **GSC/GA4-интеграция** (index coverage, orphan pages, log-file analysis — п. 5 дифференциации §6.4 каталога; требует OAuth, возможна только после аккаунтов Фазы 3а). Dev-утилиты и backlink-аудит — вне скоупа (утилиты запрещены условием 6; backlink требует внешних данных — отдельное решение).

## 7. Риски и открытые вопросы

**Риски:**

1. **Фильтр Яндекса за малополезный контент** ударит по общему домену, т.е. и по А. Митигация: условие 4 (никаких тонких страниц, запуск волнами), контент пишется до публикации инструмента.
2. **Каннибализация на широких запросах** при нарушении редакционной дисциплины. Митигация: правила §4 и артефакт `docs/seo-dev/page-checklist.md` (§4.7) в PR-review каждой новой страницы; аудит пересечений каждые 6–9 мес.
3. **Тяжесть Lighthouse/Chrome в воркере:** память/CPU, зависшие прогоны. Митигация: отдельный процесс `worker-seo.ts`, concurrency 1–2, таймаут на job, exponential backoff ≥30 с (иначе attempts = шторм запросов).
4. **SSRF/абьюз формы Б** (форма без регистрации дергает произвольные URL). Митигация встроена в план: SSRF-guard/captcha/rate-limit — пункт DoD Фазы 1, реализуются в Б и переиспользуются А; внешняя зависимость от «Фазы 2 юр-плана» снята (иначе при задержке юр-плана Б либо запускался бы уязвимым, либо стоял).
5. **Достоверность цен rechecker:** часть цифр каталога `(unverified)` — перед фиксацией `/seo/pricing/` сверить `/tools` и `/pricing` конкурента вручную.
6. **Расхождение severity:** enum `critical/major/minor` может неточно лечь на SEO-категории; принято решение не менять enum и класть специфику в `evidence.extra` — пересмотреть, если появится потребность в SQL-агрегации по SEO-категориям.
7. **Соло-разработчик и контент:** 1500–2000 знаков × ~15 страниц + FAQ — узкое горлышко запуска; волны 1а/1б сформированы под это.

**Открытые вопросы:**

1. Финальные цены и **модель расчётов** Б: платёж 20–50 ₽/мес почти съедается комиссией эквайринга, копировать цены rechecker без его механики нельзя. До фиксации `/seo/pricing/` выбрать: предоплатный баланс (механика rechecker: пополнение 100–4000 ₽, бонус 50 ₽, списания токенами) / только бандл 199 ₽/мес / годовая оплата с дисконтом — и посчитать unit-экономику монитора за 20 ₽ с учётом комиссии YooKassa. Отдельно: сетка rechecker (20–99 ₽/мес) или премиум-позиционирование за счёт глубины CWV/a11y?
2. Бандл-тариф «юр 700 ₽ + полный SEO-аудит» — решить после появления аккаунтов и данных кросс-перетока (Фаза 3а/3б).
3. ~~Единый аккаунт: вводить в Фазе 3 вместе с мониторингом или раньше?~~ **Решено:** выделен в Фазу 3а (аккаунт + история сканов + биллинг подписок) до мониторинга Фазы 3б.
4. Экспресс-аудит: сколько чеков в бесплатном слое против upsell (rechecker отдаёт все 28 бесплатно на 1 странице — вероятно, повторить)?
5. Нужна ли Б собственная лента «последних проверок» / публичная статистика Рунета как линк-магнит (паттерн rechecker `/stats`) — потенциальная волна 2 контента. До решения этого вопроса блок статистики в спецификацию хаба **не входит** (§3.1).
6. CrUX API-ключ и квоты — проверить доступность из РФ.
