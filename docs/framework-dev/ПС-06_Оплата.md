# ПС-06 · Оплата

## 0. Контекст решения (для владельца)

Рассматривался Alfa API «Работа с самозанятыми» — по документации это сервис **выплат** самозанятым исполнителям (онбординг, реестры выплат, регистрация их дохода в ФНС). Направление денег обратное нашему: там компания платит самозанятым. Для приёма 700 ₽ от клиентов картой он не предназначен.

Решение: абстракция `PaymentProvider` + реализация **ЮKassa** по умолчанию (работает с самозанятыми, редирект-сценарий, webhook, фискализация). Смена провайдера (Продамус, Т-Касса при переходе на ИП) — новая реализация интерфейса, остальной код не трогается.

**Фактчек владельцу до релиза:** условия ЮKassa для самозанятых и механику чека НПД (автопередача в «Мой налог» или ручное формирование чека) уточнить при подключении магазина — зафиксировать в этом документе. Код от этого не зависит: поле `receiptUrl` опционально.

## 1. Интерфейс

```ts
interface PaymentProvider {
  name: string;
  create(p: {
    scanId: string; amountKopecks: number; email: string; returnUrl: string;
  }): Promise<{ externalId: string; confirmationUrl: string }>;
  verifyAndParseWebhook(req: Request): Promise<
    | { valid: true; externalId: string; status: 'succeeded' | 'canceled' | 'refunded'; raw: unknown }
    | { valid: false }
  >;
}
```

Сумма — из env `PRICE_FULL_REPORT_KOPECKS=70000`, не хардкод.

## 2. YookassaProvider.create

`POST https://api.yookassa.ru/v3/payments`, Basic Auth `SHOP_ID:SECRET_KEY`, заголовок **`Idempotence-Key: {scanId}:{attempt}`** (attempt = счётчик pending-платежей скана).

```json
{
  "amount": { "value": "700.00", "currency": "RUB" },
  "capture": true,
  "confirmation": { "type": "redirect", "return_url": "{SITE_URL}/report/{scanId}?paid=1" },
  "description": "Полный отчёт проверки сайта {domain}",
  "metadata": { "scanId": "..." },
  "receipt": {
    "customer": { "email": "..." },
    "items": [{
      "description": "Информационный отчёт автоматической проверки сайта",
      "quantity": "1", "amount": { "value": "700.00", "currency": "RUB" },
      "vat_code": 1, "payment_subject": "service", "payment_mode": "full_payment"
    }]
  }
}
```

Ответ: `id` → `payments.externalId`, `confirmation.confirmation_url` → пользователю.

## 3. Состояния

```
payments: pending → succeeded | canceled → (refunded вручную через ЛК ЮKassa + admin-пометка)
scans.isPaid: false → true (только из webhook, никогда из redirect-а return_url)
```

`?paid=1` в return_url — только для UI-оптимизма («проверяем оплату…», поллинг статуса); истина приходит webhook-ом.

## 4. Webhook POST /api/webhooks/payment

1. **Подлинность**: тело парсим только после проверки. Для ЮKassa: allowlist IP-подсетей ЮKassa (константа в коде + env-переключатель `WEBHOOK_IP_CHECK=1`) **и** контрольный `GET /v3/payments/{id}` к API — статус берём из ответа API, а не из тела webhook. Это защищает от подделки даже при ошибке в allowlist.
2. **Идемпотентность**: `payments` ищем по `(provider, externalId)`; статус уже `succeeded` → ответить 200 и выйти.
3. Транзакция: `payments.status='succeeded'` + `scans.isPaid=true, email=payments.email`.
4. `queue.add('deliver', { scanId })` — PDF и письмо (ПС-07).
5. Всегда отвечать 200 при валидном событии (иначе ЮKassa ретраит); невалидное — 400 + лог.

## 5. Edge-cases

- Оплата второго пользователя по той же ссылке, когда `isPaid` уже true: `POST /api/payments` возвращает `error.code='already_paid'`, UI показывает «отчёт уже открыт».
- Скан удалён ретенцией, а платёж пришёл (гонка): webhook логирует ошибку уровня error, деньги видны в admin — возврат вручную.
- `canceled`: payments обновляется, UI по поллингу показывает «оплата не прошла» + кнопка повторить.
- Возвраты — вне MVP-кода: вручную в кабинете ЮKassa, в admin проставляется `refunded` (кнопка).

## 6. Тестирование

Песочница ЮKassa: тестовый магазин, тестовые карты. E2e-сценарий фазы 4: создать скан на фикстурном сайте → оплатить тестовой картой → получить webhook → отчёт открылся → письмо ушло (SMTP-заглушка smtp4dev в dev-compose).
