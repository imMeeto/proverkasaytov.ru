# ПС-01 · Архитектура

Архитектура наследует проверенные паттерны trava.pro: Next.js-монолит + отдельный BullMQ-воркер + Redis pub/sub для realtime. Отличие одно: воркер несёт Playwright и потому живёт в собственном Docker-образе.

## 1. Компоненты

```
[Браузер клиента]
      │ HTTPS
[Caddy / nginx]  ← TLS, gzip, rate-limit первого уровня
      │
[app: Next.js 16] ── API Routes + страницы + SSE
      │        │
      │        ├── PostgreSQL 17  (scans, check_results, payments, email_log)
      │        └── Redis          (BullMQ, rate-limit, pub/sub)
      │
[worker: Node + Playwright/Chromium]
      │  ├── краулит проверяемые сайты (наружу, через SSRF-фильтр)
      │  ├── ходит в GeoIP (локальная база GeoLite2)
      │  ├── ходит в реестр РКН (pd.rkn.gov.ru)
      │  ├── рендерит PDF отчёта (свой же /report/[id]/print)
      │  └── шлёт email (SMTP)
      │
[Yandex S3] — готовые PDF (private bucket, presigned URL)
[ЮKassa]   — платежи, webhook → app
```

## 2. Поток данных (happy path)

1. Пользователь: URL + 2 чекбокса + SmartCaptcha → `POST /api/scan`.
2. App: Zod-валидация → SSRF-предпроверка URL (ПС-08 §2) → rate-limit → дедуп (тот же домен, done-скан младше 60 мин → вернуть его id) → INSERT `scans(status='queued')` → `queue.add('scan', {scanId})` → ответ `{scanId}`.
3. Клиент редиректится на `/report/[scanId]`, открывает SSE `/api/events/scan/[scanId]`.
4. Воркер: берёт job → `status='running'` → пайплайн ПС-04 → пишет `check_results`, считает score → `status='done'` → `PUBLISH scan:{id} {"stage":"done"}`.
5. App получает publish через singleton Redis-subscriber → EventEmitter fan-out → SSE пушит клиенту (паттерн `attachSubscriberHandlers()` из trava.pro).
6. Страница отчёта перезапрашивает `GET /api/scans/[id]` → рендер: свободная часть + блюр.
7. Оплата: email + кнопка → `POST /api/payments` → redirect на ЮKassa → webhook `payment.succeeded` → `scans.is_paid=true` → job `deliver` → воркер: PDF → S3 → email.

    /api/events/scan/[id]/route.ts
    /api/payments/route.ts
    /api/webhooks/payment/route.ts
  /server
    /db                   # drizzle схема, клиент, миграции
    /queue                # bullmq подключение, имена очередей
    /checks               # 18 чеков, по файлу на чек + registry.ts
    /crawler              # playwright-пайплайн (используется только воркером)

## 4. Границы ответственности

- **App не запускает Playwright никогда.** Всё тяжёлое — в воркере.
- **Воркер не отвечает на HTTP.** Общение только через очередь и pub/sub.
- Чеки (`/server/checks`) — чистые функции без I/O: принимают собранный `ScanContext`, возвращают результат. Весь I/O (краулинг, GeoIP, РКН) делает пайплайн до них и кладёт данные в контекст. Исключение: `rkn_registry` — сетевой, вынесен в пайплайн как отдельный шаг.
- SSE-канал переиспользует паттерн trava.pro: один Redis-subscriber на процесс, fan-out через EventEmitter, никаких подписок на каждый запрос.

## 5. Очереди BullMQ

| Очередь | Job | Concurrency | Attempts / backoff |
|---|---|---|---|
| `scan` | `{scanId}` | 2 | 2 / exponential, delay ≥ 30 s |
| `deliver` | `{scanId}` | 2 | 3 / exponential, delay ≥ 30 s |

Job `scan` идемпотентен: перед стартом проверяет `status`; если уже `done` — выходит. Failed job после исчерпания попыток → `scans.status='failed'`, `error` заполняется, SSE шлёт `{"stage":"failed"}`, пользователь видит честное «не удалось проверить, попробуйте позже» и форму повторного запуска.
