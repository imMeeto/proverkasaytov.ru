# ПС-03 · Движок проверок — спецификация 21 чека (v2)

Главный документ проекта. **v2: +3 чека и правки по итогам правового ресёрча — источник фактуры ПС-11 (срез 13.07.2026); при расхождении норм/сумм ПС-11 главнее.** Каждый чек — отдельный файл в `/server/checks/`, регистрация в `registry.ts`, тесты на HTML-фикстурах (минимум pass- и fail-фикстура на чек).

## 1. Контракт чека

```ts
interface CheckDefinition {
  id: string;
  title: string;                       // 'Согласие на обработку ПДн в формах'
  category: 'infra' | 'docs' | 'forms' | 'cookies' | 'auth' | 'registry';
  severity: 'critical' | 'major' | 'minor';
  lawRef: string;                      // 'ст. 6, ст. 9 152-ФЗ'
  run(ctx: ScanContext): CheckResult;  // чистая функция, без I/O
}

interface CheckResult {
  status: 'pass' | 'warn' | 'fail' | 'unable' | 'not_applicable';
  evidence: Evidence[];
}
```

Тексты «что нашли / что делать / сниппет» и суммы штрафов живут не в коде чека, а в `checks/texts.ts` — словарь по `checkId` с полем `verifiedAt: '2026-07-13'` (правовая база — ПС-11 §7). Формулировки правятся без перекомпиляции логики.

## 2. ScanContext — что пайплайн отдаёт чекам

```ts
interface ScanContext {
  targetUrl: string;
  pages: PageSnapshot[];        // до 10 страниц
  network: NetworkRequest[];    // ВСЕ запросы всех страниц ДО взаимодействия
  ssl: SslInfo;                 // рукопожатие с корневым доменом
  geo: GeoInfo;                 // страна + ASN/провайдер (GeoLite2-Country + GeoLite2-ASN)
  rkn: RknLookup;               // реестр операторов ПДн (сетевой шаг пайплайна)
  hostingRegistry: HostingLookup; // реестр провайдеров хостинга (сетевой шаг, ПС-04 §6)
  innFound: string | null;
}

interface PageSnapshot {
  url: string;
  status: number;
  html: string;                 // после рендера JS
  text: string;                 // видимый текст
  forms: FormInfo[];            // извлечённые формы, см. ПС-04 §5
  links: { href: string; anchor: string }[];
  cookiesSet: string[];         // имена кук, поставленных до взаимодействия
}

interface NetworkRequest {
  pageUrl: string;
  requestUrl: string;
  domain: string;               // eTLD+1
  type: string;                 // script | xhr | font | image | ...
  beforeConsent: true;          // в MVP собираем только до взаимодействия
}
```

## 3. Скоринг

`score = max(0, 100 − Σ penalty)`.

| Статус \ severity | critical | major | minor |
|---|---|---|---|
| fail | −14 | −8 | −3 |
| warn | −4 | −3 | −1 |
| pass / unable / not_applicable | 0 | 0 | 0 |

Светофор: **≥ 80 зелёный** («серьёзных проблем не найдено»), **50–79 жёлтый** («есть нарушения»), **< 50 красный** («много нарушений»). `unable` показывается отдельной строкой «не удалось проверить автоматически» со ссылкой на ручную проверку — балл не снижает.

Свободная часть отчёта: score + счётчики + **два fail с наибольшим penalty** полностью; остальные fail/warn — заголовок и статус, тело под блюром.

## 4. Группа A · Инфраструктура

### A1 `https_ssl` — HTTPS и сертификат · major · ст. 19 152-ФЗ (меры защиты ПДн)
Данные: `ctx.ssl` (пайплайн делает TLS-рукопожатие, забирает цепочку, notAfter).
- **fail**: сайт не отвечает по HTTPS, или сертификат невалиден/просрочен, или http:// не редиректит на https://.
- **warn**: сертификат истекает < 14 дней.
- **pass**: HTTPS валиден, редирект с http есть.
Evidence: issuer, validTo, daysLeft.

### A2 `hosting_geo` — локализация хостинга · major · ч. 5 ст. 18 152-ФЗ
Данные: `ctx.geo` (локальная база GeoLite2-Country по IP домена).
- **fail**: country ≠ RU.
- **pass**: country = RU.
- **unable**: IP не определился.
Обязательная оговорка в тексте отчёта: «определено по IP входной точки; если сайт за CDN/прокси, фактическое размещение БД может отличаться — проверьте у хостера». Штрафы за локализацию — ч. 8/ч. 9 ст. 13.11: 1–6 млн, повторно до 18 млн (суммы в texts.ts из ПС-11 §1.6).

### A3 `hosting_rkn_registry` — хостер в реестре провайдеров хостинга · minor · 406-ФЗ, ст. 56.2 126-ФЗ **[новый, v2]**
С 01.02.2024 услуги хостинга в РФ вправе оказывать только провайдеры из реестра РКН. Прямого штрафа владельцу сайта нет → максимум warn; но зарубежный/нереестровый хостер — маркер риска и дубль-сигнал локализации.
Данные: `ctx.geo.asn/provider` (GeoLite2-ASN) + `ctx.hostingRegistry` (сервис ПС-04 §6: XLSX-реестр с rkn.gov.ru, кэш 7 дней, матчинг через словарь `asn-hosters.ts` ASN → наименование/ИНН для топ-30 российских хостеров).
- **warn**: country ≠ RU (иностранный хостер в реестре состоять не может), либо RU-провайдер уверенно определён и в реестре не найден.
- **pass**: провайдер определён и найден в реестре.
- **unable**: ASN не сматчился со словарём, либо реестр недоступен — «проверьте вручную» + ссылка на реестр.
Ложных fail быть не должно: любое сомнение = unable.

## 5. Группа B · Документы

### B1 `privacy_policy_exists` — наличие политики ПДн · critical · ст. 18.1 152-ФЗ
Логика: по всем `pages[].links` ищем ссылки, где anchor или href матчит словарь: `политик`, `конфиденциальн`, `персональн`, `privacy`, `policy`, `personal-data`, `pd`. Найденная страница должна быть в `pages` (краулер обязан её включить, ПС-04 §3) и отдавать 200 с текстом > 1000 символов.
- **fail**: ссылок нет ни на одной странице, либо страница битая/пустая.
- **pass**: найдена и открывается.
Evidence: URL политики. Санкция в texts.ts: ч. 3 ст. 13.11.

### B2 `privacy_policy_content` — содержание политики · major · ч. 2 ст. 18.1 152-ФЗ
Работает только если B1 = pass, иначе `not_applicable`. Скриптовый режим (без LLM): по тексту страницы политики ищем 8 обязательных смысловых блоков, каждый — словарь ключевых фраз (все формы через стемминг/lowercase):
1. наименование оператора: `оператор` + (ИНН|ОГРН|наименование)
2. цели обработки: `цел` + `обработк`
3. правовые основания: `правов` + `основани`
4. категории/перечень данных: `категори|перечень` + `данн`
5. сроки хранения: `срок` + `хранени`
6. права субъекта: `права субъект|субъект персональных данных вправе`
7. уничтожение/прекращение обработки: `уничтожени|прекращени обработк`
8. передача третьим лицам / поручение: `третьим лицам|поручени`
- **pass**: ≥ 7 блоков. **warn**: 5–6. **fail**: < 5 («документ, вероятно, шаблонный»).
Evidence: список ненайденных блоков. **v2, текст отчёта:** добавить требование актуальности — по практике автомониторинга РКН 2026 устаревшая политика приравнивается к нарушению; рекомендация пересматривать документ при каждом изменении форм/сервисов. Пометка: «автоматическая эвристика; окончательную оценку даёт юрист». Точка расширения LLM — §11.

### B3 `operator_details` — реквизиты оператора/продавца · major · ст. 9 ЗОЗПП, ст. 10 149-ФЗ
Логика: по тексту всех страниц (приоритет — футер, /contacts, /requisites):
- ИНН: regex `\b\d{10}\b|\b\d{12}\b` **с валидацией контрольных цифр** (алгоритм ФНС, обязательно — иначе ловим телефоны). Найденный валидный ИНН → `ctx.innFound`.
- Наименование/ОГРН(ИП): `ООО|ИП|АО|самозанят` рядом с ИНН, либо `\b\d{13}\b|\b\d{15}\b` с валидацией ОГРН.
- Контакт: email ИЛИ телефон.
- **pass**: ИНН + наименование + контакт. **warn**: часть найдена. **fail**: ничего.

### B4 `offer_exists` — публичная оферта · minor · ст. 437 ГК РФ
Применимость: сначала детект коммерции — на страницах есть цены (`\d[\d\s]{2,}\s?(₽|руб)`) + кнопки покупки (`купить|заказать|оплатить|в корзину|оформить`). Нет коммерции → `not_applicable`.
- **fail**: коммерция есть, ссылки `оферт|договор|условия` нет.
- **pass**: ссылка есть и страница открывается.

### B5 `age_rating` — возрастная маркировка · minor · 436-ФЗ
Логика: regex `(^|\s)(0|6|12|16|18)\s?\+` в html главной (шапка/футер).
- **warn**: не найдена (практика применения к обычным сайтам неоднородна — не fail).
- **pass**: найдена.

## 6. Группа C · Формы

Извлечение форм — в краулере (ПС-04 §5). Форма считается «собирающей ПДн», если содержит input с type `tel|email` или name/placeholder из словаря: `name, имя, phone, телефон, email, почта, фамили, address, адрес`.

### C1 `forms_consent_checkbox` — согласие в формах · critical · ст. 6, ст. 9 152-ФЗ **[правки v2]**
Для каждой ПДн-формы проверяем три условия (любое нарушение → fail):
1. Внутри формы есть `input[type=checkbox]`, в тексте label/родителя — `согла` + (`персональн|обработк`). Нет чекбокса → **fail**.
2. **[v2]** Чекбокс не предустановлен: атрибут `checked` (или `aria-checked="true"` при рендере) у чекбокса согласия → **fail** («предустановленная галочка — не согласие», подтверждено предписаниями РКН, ПС-11 §1.3).
3. **[v2]** Формулировка «согласия действием»: внутри формы текст `отправляя (форму|заявку)|нажимая (кнопку|отправить)` + `соглаша|согласие` при отсутствии чекбокса → **fail** (РКН признаёт только активный чекбокс).
- **pass**: у всех ПДн-форм непредустановленный чекбокс. **not_applicable**: ПДн-форм нет.
Evidence по каждой проблемной форме: pageUrl, selector, поля, тип нарушения (нет чекбокса / предустановлен / «отправляя…»). Сниппет платной части — label с required-чекбоксом. Санкция: ч. 2 ст. 13.11 — 300–700 тыс., повторно до 1,5 млн.

### C2 `forms_policy_link` — ссылка на политику у формы · minor · ст. 18.1 152-ФЗ
Внутри формы или её родительского контейнера (до 3 уровней вверх) есть `<a>` на страницу политики (из B1).
- **warn**: у ≥ 1 ПДн-формы нет. **pass**: у всех есть.

### C3 `newsletter_consent` — согласие на рассылку · minor · ст. 18 38-ФЗ «О рекламе»
Детект форм подписки: единственное поле email + рядом текст `подпис|рассылк|новост`.
- **fail**: форма подписки без отдельного чекбокса с текстом `реклам|рассылк`.
- **pass** / **not_applicable**. Санкция: ч. 1 ст. 14.3 — до 500 тыс.

### C4 `forms_external_action` — куда уходят данные форм · major · ч. 3 ст. 6 152-ФЗ
Логика MVP: атрибут `action` каждой ПДн-формы. Если домен action ≠ eTLD+1 сайта и не в allowlist легальных форм-сервисов с РФ-обработкой — fail. Плюс матчинг известных сборщиков форм по network-запросам.
- **fail**: данные форм уходят на чужой домен.
- **pass**: все action свои/относительные.
Оговорка в отчёте: js-отправку без атрибута action MVP не детектирует.

### C5 `combined_consent` — согласие объединено с другими документами · major · ч. 1 ст. 9 152-ФЗ (ред. 156-ФЗ), с 01.09.2025 **[новый, v2]**
С 01.09.2025 согласие на обработку ПДн оформляется отдельно: его нельзя объединять с офертой, условиями, пользовательским соглашением; разные виды согласий не совмещаются в одном чекбоксе.
Логика: для каждого чекбокса согласия (найденного в C1) анализ текста label/родителя:
- содержит `соглас` + `персональн|обработк` **И** одновременно `оферт|условия использования|пользовательск(ое|им) соглашени|правила сервиса|договор` → **fail** (согласие склеено с договором);
- содержит `соглас` + `персональн|обработк` **И** `реклам|рассылк` в одном чекбоксе → **fail** (два вида согласий в одном).
- **pass**: все чекбоксы «чистые». **not_applicable**: чекбоксов согласия нет (тогда работает C1).
**Защита от false positive:** слово «Политика» и ссылка на неё внутри текста согласия — норма, НЕ триггер; триггерят только документы-договоры. Фраза «в соответствии с Политикой обработки персональных данных» — pass.
Evidence: pageUrl, selector, полный текст label. Сниппет платной части: два раздельных чекбокса.

## 7. Группа D · Cookie и трекеры (поведенческие)

Ключевое: `ctx.network` собран **до любого взаимодействия** со страницей — робот не кликал «Принять».

Словарь трекеров `trackers.ts` (расширяемый), формат `{ domain, name, foreign: boolean }`:
`mc.yandex.ru` (Метрика, RU) · `google-analytics.com`, `googletagmanager.com` (GA, foreign) · `vk.com/rtrg`, `top.mail.ru`, `top-fwz1.mail.ru` (VK/Top, RU) · `connect.facebook.net` (Meta Pixel, foreign) · `analytics.tiktok.com` (foreign) · `criteo.com` (foreign) · `yandex.ru/ads`, `an.yandex.ru` (RU).

### D1 `cookie_banner` — наличие cookie-баннера · minor · ст. 9 152-ФЗ (практика РКН)
Логика: в html/text страниц — блок, содержащий `cookie|куки` + кнопку/ссылку `принять|соглас|ок|хорошо`, либо классы известных CMP (`cookiescript`, `cookie-consent`, `cmp-` и т.п.).
- **fail**: трекеры есть (по словарю), баннера нет.
- **warn**: трекеров нет и баннера нет (формально куки может ставить и сам сайт).
- **pass**: баннер найден.

### D2 `cookie_banner_decline` — равнозначный отказ · minor · ст. 9 152-ФЗ
Только если D1 нашёл баннер, иначе `not_applicable`. В блоке баннера ищем вторую кнопку: `отклон|отказ|только необходим|нет`.
- **warn**: только «Принять».
- **pass**: выбор есть.

### D3 `trackers_before_consent` — счётчики до согласия · critical · ст. 9 152-ФЗ **[уточнение v2]**
Логика: в `ctx.network` есть запросы к доменам из словаря трекеров — а взаимодействия не было. **Российские счётчики (Метрика, VK) — тоже нарушение**: IP и cookie-идентификаторы отнесены практикой к ПДн, запуск до согласия = сбор без согласия (ПС-11 §2).
- **fail**: есть хотя бы один. Evidence: имя трекера, домен, pageUrl.
- **pass**: до согласия трекеры молчат.
Санкция: ч. 1/ч. 2 ст. 13.11. Сниппет платной части: cookie-баннер с `startAnalytics()` после клика.

### D4 `foreign_trackers` — иностранные трекеры · major · ст. 12 152-ФЗ (трансграничная передача)
- **fail**: в network есть домены словаря с `foreign: true` (GA, Meta Pixel, TikTok, Criteo).
- **pass**: только российские или никаких.
Текст отчёта: трансграничная передача требует отдельного уведомления РКН до её начала (ПС-11 §1.7).

### D5 `foreign_resources` — иностранные шрифты, капчи, embed · minor · ст. 12 152-ФЗ, позиция РКН по IP
Словарь ресурсов: `fonts.googleapis.com`, `fonts.gstatic.com`, `www.google.com/recaptcha`, `www.gstatic.com/recaptcha`, `youtube.com/embed`, `player.vimeo.com`, `cdnjs.cloudflare.com`, `unpkg.com`, `cdn.jsdelivr.net`.
- **warn**: найдены (IP посетителей уходит за рубеж). Evidence: какие именно.
- **pass**: не найдены.
Сниппет: self-hosted `@font-face`, замена reCAPTCHA на SmartCaptcha, VK Видео/Rutube вместо YouTube.

## 8. Группа E · Авторизация и рекомендательные технологии

### E1 `foreign_auth` — вход через иностранные сервисы · critical · 149-ФЗ (требование с 01.12.2023) + **ст. 13.55 КоАП (199-ФЗ от 26.06.2026, в силе с 07.07.2026)** **[норма зафиксирована, v2]**
Разрешённые способы авторизации российских пользователей: номер телефона РФ, ЕСИА (Госуслуги), ЕБС, российские сервисы (VK ID, Яндекс ID, Сбер ID, Mail ID).
Логика — детектим только OAuth-механику иностранных провайдеров:
- скрипты/iframe: `accounts.google.com/gsi`, `appleid.apple.com/auth`, `connect.facebook.net` (Login), `github.com/login/oauth`, `login.microsoftonline.com`, `discord.com/oauth`;
- кнопки: текст `войти через|sign in with|continue with` + `Google|Apple|Facebook|GitHub|Microsoft|Discord`.
- **fail**: найдено.
- **pass**: авторизация есть, иностранной механики нет. **not_applicable**: форм логина на сайте нет.
**Защита от false positive (обязательно):** обычная форма `input[type=email]` + пароль — НЕ нарушение, даже если пользователь введёт @gmail.com; поле email в формах обратной связи — НЕ нарушение. Триггер — только механизм делегированной авторизации.
Санкция (texts.ts): юрлица 500–700 тыс., повторно 1,4–2 млн; должностные/ИП 30–50 тыс.; дела рассматривает РКН. Дата в тексте отчёта: «штрафы действуют с 07.07.2026».

### E2 `recsys_rules` — правила рекомендательных технологий · minor · ст. 10.2-1 149-ФЗ (штрафы — 199-ФЗ) **[новый, v2]**
Сайт, применяющий рекомендательные алгоритмы, обязан публиковать правила их применения. Штрафы введены тем же 199-ФЗ, для юрлиц при повторном нарушении — до 2,8 млн.
Логика (консервативная эвристика):
- детект рекомендательных блоков: текст `вам может понравиться|рекомендуем вам|похожие товары|с этим (товаром )?покупают|персональные рекомендации|специально для вас` **И** рядом карточная разметка (≥ 2 однотипных ссылок-карточек) или классы `recommend|similar|related-products`;
- при детекте — поиск ссылки/страницы `рекомендательн` + `технолог|правил`.
- **warn**: блоки найдены, правил нет.
- **pass**: блоки найдены, правила опубликованы. **not_applicable**: рекомендательных блоков не найдено (дефолт для большинства малых сайтов).
**Защита от false positive:** слово «рекомендуем» в тексте статьи без карточного блока — не триггер. В тексте отчёта: «правила обязательны только при алгоритмических рекомендациях; статические подборки, собранные вручную, под норму не подпадают — если подборка ручная, отметьте это и проигнорируйте пункт».

## 9. Группа F · Внешние реестры

### F1 `rkn_registry` — реестр операторов ПДн РКН · critical · ст. 22 152-ФЗ **[текст усилен, v2]**
Сетевой шаг пайплайна (не чистая функция): если `ctx.innFound` есть → запрос к порталу pd.rkn.gov.ru (поиск по ИНН, парсинг ответа; таймаут 15 с; ретрай 1).
- **fail**: ИНН найден на сайте, в реестре записи нет — и при этом на сайте есть ПДн-формы (сбор данных ведётся).
- **pass**: запись в реестре найдена.
- **unable**: ИНН не найден на сайте, либо реестр недоступен/изменил разметку → текст «проверьте вручную» + прямая ссылка на реестр. Парсер обязан деградировать в `unable`, а не в ложный fail.
Текст отчёта (v2): РКН сверяет сайты с реестром операторов **автоматически** (система мониторинга, ПС-11 §1.10); сайт собирает данные + записи нет = типовое основание предписания со сроком устранения 10 рабочих дней. Санкция ч. 10 ст. 13.11: 100–300 тыс.
