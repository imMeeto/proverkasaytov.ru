# ПС-10 · Развёртывание

Наследует схему trava.pro (Docker Compose на российском VPS, WAF отдельно, secrets в env), но **на своём отдельном сервере** — не рядом с trava.pro. Причины: Playwright прожорлив по RAM; юридически чище держать данные клиентов сервиса изолированно от данных TRAVA.

## 1. Хостинг

- VPS в РФ (Timeweb Cloud / Selectel / sweb — как у trava.pro). Минимум под MVP: **4 vCPU / 8 GB RAM / 60 GB NVMe**, Ubuntu 24.04. 8 GB, потому что воркер с Chromium при concurrency 2 просит ~2 GB, плюс PG+Redis+app.
- Домен `.ru` на владельца (напоминание: с 01.09.2026 продление — через идентификацию на Госуслугах).

## 2. Контейнеры (docker-compose.prod.yml)

| Сервис | Образ | Порты | Сеть |
|---|---|---|---|
| `app` | Dockerfile.app (node:22-slim, next start) | :3000 только внутрь | frontend + backend |
| `worker` | Dockerfile.worker (playwright:1.x-noble) | — | backend + egress |
| `postgres` | postgres:17 | — | backend |
| `redis` | redis:7 | — | backend |
| `caddy` | caddy:2 | :80/:443 наружу | frontend |

- `app` слушает только внутреннюю сеть; наружу выходит через Caddy (TLS Let's Encrypt автоматом, HSTS, gzip).
- БД и Redis портов наружу не публикуют вообще.
- Env — файл `/opt/<app>/.env`, права `600`, владелец root. В образы секреты не зашиваются.

## 3. WAF / проксирование

Вариант A (проще, для старта): Caddy на том же VPS + fail2ban + rate-limit Caddy — достаточно для MVP.

Вариант B (как trava.pro, когда пойдёт трафик): вынести Nginx + CrowdSec на отдельный Yandex VPS, канал WAF→app по WireGuard, app слушает только `10.8.0.2:3000`, изоляция бэкенда через `X-Proxy-Secret`. Документ не требует B на запуске — заложить возможность, включить при нагрузке.

## 4. Сетевая изоляция воркера (критично, связка с ПС-08 §4)

Две Docker-сети: `backend` (app, worker, postgres, redis) и `egress` (только worker). Правила на хосте (nft/iptables) для подсети контейнера `worker`:
- **разрешить** исходящие на публичные IP (краулинг, РКН, S3, ЮKassa, SMTP);
- **разрешить** на `app`, `postgres`, `redis` по внутренним адресам;
- **запретить** любые исходящие в `10/8, 172.16/12, 192.168/16, 169.254/16` и прочие приватные, кроме адресов трёх сервисов выше.

Так SSRF-запрос, проскочивший route-фильтр, упрётся в firewall. Пояс и подтяжки.

## 5. Деплой

- Ручной: `docker compose -f docker-compose.prod.yml build && up -d` (паттерн trava.pro).
- Авто (позже): self-hosted GitHub Actions runner на VPS, деплой по push в `main`. Секреты — в env на сервере, не в Actions.
- Миграции БД — отдельный шаг перед `up`: `drizzle-kit migrate` (job в compose или руками), не в рантайме app.
- GeoLite2-база: скачивается при сборке образа воркера или монтируется томом; обновление — cron раз в месяц.
- Playwright-браузеры уже в базовом образе — заново не ставим.

## 6. Наблюдаемость и надёжность

- Логи: pino → stdout → `docker compose logs` / драйвер json-file с ротацией (max-size 10m, max-file 5).
- Healthcheck-эндпоинт `/api/health`: проверяет PG, Redis, длину очереди — для Caddy/мониторинга.
- Аптайм-мониторинг снаружи (UptimeRobot/аналог РФ) на `/` и `/api/health`.
- Бэкапы PostgreSQL: `pg_dump` ежедневно cron-job → шифрование → выгрузка в отдельный S3-bucket, храним 30 копий. Проверка восстановления — раз в месяц вручную.
- Redis — не бэкапим (очередь и кэш эфемерны; при потере сканы просто перезапускаются пользователем).
- Алерты: длина очереди `scan` > 50 или `worker` disconnected > 3 раз за час → уведомление владельцу (email/Telegram).

## 7. Порядок первого развёртывания

1. VPS + домен + DNS A-запись на VPS.
2. `.env` заполнен (все ключи из ПС-01 §6 + платёжка + SMTP + S3).
3. `docker compose build && up -d`; миграции; проверить `/api/health`.
4. Caddy выдал TLS; открывается `https://домен`.
5. Тестовый скан фикстурного сайта проходит end-to-end.
6. Юрчек-лист ПС-09 §4 закрыт → самоскан 100/100 → публичный запуск на своих (тихий старт из ЛБ-06).
