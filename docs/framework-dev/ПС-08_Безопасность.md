# ПС-08 · Безопасность

Обязателен к прочтению **до** написания краулера. Наш робот по команде пользователя ходит по произвольным URL — это готовый SSRF-вектор, если не закрыть.

## 1. Модель угроз

1. **SSRF**: пользователь подсовывает `http://192.168.1.1/`, `http://localhost:6379/`, `http://169.254.169.254/` (metadata-сервисы облаков), или домен, резолвящийся в приватный IP, или редирект туда же. Цель — заставить наш воркер сканировать нашу же инфраструктуру.
2. **Абьюз ресурсов**: массовые бесплатные сканы (DoS воркера), скан чужих сайтов ради вреда.
3. **Подделка оплаты**: фейковый webhook (закрыто в ПС-06 §4 контрольным запросом к API).
4. **Перебор отчётов**: UUID v4 достаточно энтропии; дополнительно — noindex и отсутствие листинга.

## 2. SSRF: предпроверка в API (быстрая, до постановки в очередь)

`security/ssrf.ts` → `assertPublicUrl(url)`:
- схема только `http|https`; порт только 80/443/пусто; в URL нет credentials (`user:pass@`).
- hostname не IP-литерал из запрещённых диапазонов и не `localhost|*.local|*.internal`.
- `dns.resolve4/6` → **все** полученные адреса обязаны быть публичными. Запрещённые диапазоны: `0.0.0.0/8, 10/8, 100.64/10, 127/8, 169.254/16, 172.16/12, 192.168/16, 198.18/15, ::1/128, fc00::/7, fe80::/10` + IPv4-mapped IPv6.
- hostname ≠ наш собственный домен и ≠ IP нашего VPS.

## 3. SSRF: финальная проверка в воркере

DNS может измениться между постановкой в очередь и сканом (DNS rebinding). Воркер перед `goto` резолвит заново и проверяет ещё раз. Редиректы: `page.on('response')` — если конечный URL сменил хост, новый хост прогоняется через `assertPublicUrl`; редирект в приватную зону → скан прерывается со статусом `failed('unsafe_redirect')`.

## 4. SSRF: route-фильтр внутри страницы

Проверяемый сайт может сам дёргать `http://10.0.0.5/...` из своего JS. Фильтр на каждый под-запрос:

```ts
await page.route('**/*', route => {
  const u = new URL(route.request().url());
  if (!['http:', 'https:'].includes(u.protocol)) return route.abort();
  if (isForbiddenHostname(u.hostname)) return route.abort();  // литералы, localhost, .local
  return route.continue();
});
```

Резолвить DNS каждого под-запроса дорого — поэтому **обязательна сетевая изоляция** (пояс и подтяжки): контейнер воркера в отдельной Docker-сети; iptables/nft на хосте запрещают контейнеру воркера исходящие в RFC1918 и link-local, кроме адресов `app`, `postgres`, `redis` (ПС-10 §4). Тогда даже пропущенный фильтром запрос упрётся в сеть.

## 5. Антиабьюз

- SmartCaptcha на создание скана — серверная верификация токена, без валидного токена 403.
- Rate-limit (Redis, паттерн `redisCache.rateLimit()`): `scan:create` — 5/сутки/IP и 2/час/IP; `payments:create` — 10/сутки/IP; `pdf-url` — 20/сутки/IP. Ответ 429 с человекочитаемым текстом.
- Дедуп доменов (60 мин) — экономит воркер и режет спам-прогоны.
- Глобальный предохранитель: длина очереди `scan` > 50 → API отвечает «высокая нагрузка, попробуйте через несколько минут» (503) — честнее, чем копить часовую очередь.
- Один домен — не более 3 сканов/сутки суммарно от всех IP (защита чужого сайта от долбёжки нашим же роботом).

## 6. Этика робота

- User-Agent: `Mozilla/5.0 (compatible; {SITE_NAME}Bot/1.0; +{SITE_URL}/bot)`.
- Страница `/bot`: кто мы, зачем приходим (по заявке владельца сайта), лимиты (≤10 страниц, ≤1 раз/час), контакт для блокировки домена. Опция `blocked_domains` в БД-конфиге — по письму вносим домен, API отказывает в скане.
- Пауза 500 мс между страницами одного сайта; параллельно страницы одного сайта не грузим.
- Чекбокс «являюсь владельцем/уполномочен» — фиксируется в `scans.meta.consents` вместе с временем; текст оферты это дублирует.

## 7. Прикладная гигиена

- Zod `.strict()` всюду; UUID-параметры валидируются до запроса в БД.
- Заголовки: CSP (`default-src 'self'`; `connect-src 'self' https://mc.yandex.ru https://smartcaptcha.yandexcloud.net`; `frame-src` для SmartCaptcha и виджета оплаты), `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, HSTS на прокси.
- Печатная версия — только по HMAC-токену; сравнение `crypto.timingSafeEqual`.
- В логи и Sentry-подобные сервисы не пишем: email целиком, тела webhook с ПДн. IP пользователя живёт только в Redis-ключах rate-limit с TTL ≤ 24 ч (SHA-256 от IP + соль из env).
- `/admin`: bcrypt-пароль из env, httpOnly-кука на 12 ч, rate-limit 5 попыток/час.
- Зависимости: lockfile, `pnpm audit` в CI; Playwright и Next обновляются осознанно, не автоматом.
