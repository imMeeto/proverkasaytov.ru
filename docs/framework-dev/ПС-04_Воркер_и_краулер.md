# ПС-04 · Воркер и краулер

Воркер — отдельный Node-процесс (`worker.ts`), образ на базе `mcr.microsoft.com/playwright:<текущий v1.x>-noble`. Один экземпляр браузера Chromium на процесс, новый `BrowserContext` на каждый скан (изоляция кук), concurrency = 2.

## 1. Жизненный цикл job `scan`

```
получить job {scanId} → прочитать scans → status='running', started_at
→ SSRF-финальная проверка URL (ПС-08)
→ ШАГ 1: TLS-инфо + GeoIP           publish {stage:'infra'}
→ ШАГ 2: выбор страниц               publish {stage:'pages', total}
→ ШАГ 3: обход страниц (цикл)        publish {stage:'crawl', page, i, total}
→ ШАГ 4: извлечение ИНН → реестр РКН publish {stage:'registry'}
→ ШАГ 5: прогон 18 чеков             publish {stage:'checks'}
→ ШАГ 6: скоринг, upsert результатов, status='done', finished_at
→ publish {stage:'done', score}
```

Любая необработанная ошибка → attempts (2, exponential backoff ≥ 30 s) → после исчерпания `status='failed'`, publish `{stage:'failed'}`. Глобальный таймаут скана `SCAN_TOTAL_TIMEOUT_MS` (150 с) через `Promise.race` — по истечении контекст браузера принудительно закрывается.

## 2. Выбор страниц (max `SCAN_MAX_PAGES` = 10)

1. Главная — всегда.
2. `robots.txt` → найти sitemap; распарсить sitemap (до 200 URL, только тот же eTLD+1).
3. Собрать кандидатов: ссылки главной + sitemap. Приоритизация по словарю путей/анкоров, по убыванию веса:
   `privacy|policy|конфиденц|персональн` (обязательно, если есть — нужна чеку B2) → `contact|контакт` → `requisites|реквизит|about|о нас|о компании` → `cart|basket|корзин|checkout|заказ` → `delivery|доставк|возврат|return` → `price|цен|услуг|catalog|каталог` → прочие.
4. Дедуп по нормализованному URL, отсечение якорей/utm, только http(s), только свой домен (поддомены www — ок).

**[Правовой ресёрч 07.2026] Уважение robots.txt и стоп-КИИ.** Краулер соблюдает `Disallow` и `Crawl-delay` из `robots.txt` проверяемого сайта (не только вытаскивает из него sitemap) — страницы под `Disallow` не обходятся. Категорически не обходить формы авторизации, CAPTCHA, антибот-защиту. Стоп-флаг для объектов КИИ (банки, госорганы, операторы связи, энергетика — по словарю доменов/ASN): скан отклоняется на входе (ст. 274.1 УК). См. ПС-08 §8, docs/legal-research-2026-07.md.

## 3. Обход одной страницы

```ts
const page = await context.newPage();
const requests: NetworkRequest[] = [];
page.on('request', r => requests.push(toNetworkRequest(r, pageUrl)));
await page.route('**/*', route =>
  ssrfGuard(route.request().url()) ? route.continue() : route.abort()  // ПС-08 §4
);
await page.goto(url, { waitUntil: 'networkidle', timeout: 20_000 });
await page.waitForTimeout(2_000);       // добираем ленивые скрипты и баннеры
// НИКАКИХ кликов и скроллов — network должен остаться «до согласия»
const snapshot = await collectSnapshot(page);  // html, text, forms, links, cookies
await page.close();
```

Настройки контекста: `userAgent` из ПС-08 §6, `viewport 1366×900`, `locale: 'ru-RU'`, `serviceWorkers: 'block'`. Блокируем тяжёлое: `route.abort()` для `media` и запросов > 5 МБ. **Шрифты и скрипты не блокировать** — они нужны чекам D5/D3.

Ошибка загрузки страницы (таймаут, 5xx): страница помечается в `pagesCrawled` со статусом, скан продолжается. Если недоступна сама главная — скан `failed` с человекочитаемой ошибкой («сайт не отвечает»).

## 4. Сбор TLS и GeoIP (шаг 1, без браузера)

- TLS: `tls.connect` на 443 корневого домена → сертификат: issuer, validTo, daysLeft; отдельный HEAD на `http://` → проверка редиректа на https.
- DNS: `dns.promises.resolve4` → первый публичный IP (ПС-08 валидирует) → GeoLite2-Country lookup локальной базой (`GEOIP_DB_PATH`), без сетевых вызовов.
- ASN/провайдер: GeoLite2-ASN lookup той же механикой (`GEOIP_ASN_DB_PATH`) → `ctx.geo.asn/provider` — нужно чеку A3 (реестр хостинг-провайдеров). Словарь `asn-hosters.ts` мапит ASN топ-30 российских хостеров на наименование/ИНН.

## 5. Извлечение форм (`collectSnapshot`)

В контексте страницы через `page.evaluate`:

```ts
forms = [...document.querySelectorAll('form')].map(f => ({
  selector: cssPath(f),                      // короткий уникальный селектор
  action: f.getAttribute('action') ?? '',
  method: f.method,
  fields: [...f.querySelectorAll('input,textarea,select')].map(i => ({
    name: i.name, type: i.type, placeholder: i.placeholder ?? ''
  })),
  hasCheckbox: !!f.querySelector('input[type=checkbox]'),
  checkboxText: textAround(f.querySelector('input[type=checkbox]')), // label/родитель
  innerText: f.innerText.slice(0, 2000),
  nearbyLinks: linksWithin(f.closest('section,div') ?? f)             // для чека C2
}))
```

Также собрать: `document.documentElement.outerHTML`, `document.body.innerText`, все `<a>` (href+anchor), `document.cookie` имена.

## 6. Реестр РКН (шаг 4)

Модуль `services/rkn.ts`: обычный `fetch` (не браузер) к поисковому эндпоинту портала pd.rkn.gov.ru с ИНН; таймаут 15 с, 1 ретрай. Разметка портала нестабильна → парсер обязан при любом несоответствии ожиданиям возвращать `{ status: 'unavailable' }`, чек F1 превратит это в `unable`. Кэш результата в Redis на 24 ч по ИНН.

Модуль `services/hosting-registry.ts` (для чека A3): скачивание публичного XLSX-реестра провайдеров хостинга со страницы rkn.gov.ru (`activity/connection/register/p1578/`), парсинг (наименование + ИНН провайдера), кэш в Redis на 7 суток. Сверка: провайдер из `asn-hosters.ts` → поиск по ИНН/наименованию в реестре. Та же дисциплина деградации: файл недоступен или формат изменился → `{ status: 'unavailable' }` → чек A3 даёт `unable`, никогда не ложный warn.

## 7. Прогресс и SSE-события

Publish в Redis-канал `scan:{scanId}`, JSON `{stage, ...}`. App-процесс держит singleton-subscriber (паттерн trava.pro) и раздаёт по EventEmitter в открытые SSE-соединения. Событий немного, но пользователь должен видеть жизнь: «проверяю соединение → обхожу страницу 3 из 8 → сверяю реестр РКН → считаю балл».

## 8. Ресурсы и деградация

- Chromium ~ 300–500 МБ на активный контекст. При concurrency 2 воркеру нужно ~ 2 ГБ RAM.
- `browser.on('disconnected')` → пересоздание браузера, текущие job уходят в retry.
- Каждые 50 сканов — профилактический рестарт браузера (защита от утечек).
- Cron внутри воркера (BullMQ repeatable): ретенция ПС-02 §5, прогрев GeoIP-базы, очистка зависших `running` старше 10 минут → `failed`.
