# CLAUDE.md — корневая инструкция проекта

Кодовое имя: **lawcheck**. Продукт: веб-сервис автоматической проверки сайтов на соответствие законодательству РФ (152-ФЗ, ЗОЗПП, 436-ФЗ, требования 2026 года). Бесплатный скан → балл и 2 нарушения; полный отчёт — 700 ₽.

Бренд и домен не захардкожены: `NEXT_PUBLIC_SITE_NAME`, `NEXT_PUBLIC_SITE_URL` из env.

## Стек (зафиксирован, не менять)

- Next.js 16 (App Router) · React 19 · TypeScript strict · Tailwind · shadcn/ui
- PostgreSQL 17 + Drizzle ORM (миграции только через drizzle-kit)
- Redis: очередь BullMQ + rate-limit + pub/sub для SSE
- Воркер: отдельный Node-процесс `worker.ts` (BullMQ Worker) + Playwright/Chromium
- Zod `.strict()` на каждом входе API
- Yandex SmartCaptcha, Yandex S3 (PDF), российский SMTP

## Жёсткие конвенции

1. **Сервис обязан проходить собственную проверку на 100/100.** Поэтому в клиентском коде запрещены: Google Fonts и любые внешние шрифты (только self-hosted через `@font-face`), reCAPTCHA, иностранные CDN для скриптов, OAuth Google/Apple, любые счётчики до согласия пользователя. Нарушение этого пункта = провал задачи, каким бы удобным ни было решение.
2. Секреты только через `process.env`. Никогда в коде, никогда в клиентском бандле (кроме `NEXT_PUBLIC_*`).
3. Никакого localStorage для критичных данных; сессии не нужны (ЛК нет).
4. Все внешние вызовы (платёжка, РКН, GeoIP, SMTP) — с таймаутами и обработкой отказа. Недоступность внешнего сервиса не должна ронять скан: чек получает статус `unable`.
5. LLM в чеках не используется. Точка расширения описана в ПС-03 §11, но в MVP её не реализуем.
6. Логи — pino, структурированные. В логи не пишутся email-адреса целиком (маскировать до `a***@domain`).
7. Тексты интерфейса и отчётов — русский язык, тон нейтрально-деловой, без запугивания. Формулировки нарушений — из ПС-03 и ПС-11, не изобретать; суммы штрафов — только из texts.ts.
8. Каждый чек — чистая функция от `ScanContext` (ПС-03 §2), покрывается unit-тестом на фикстурах HTML.

## Фазы разработки и Definition of Done

**Фаза 1 — каркас.** Монорепо, docker-compose.dev (postgres, redis), Next.js app, схема БД + миграции, BullMQ очередь `scan`, пустой воркер, SSE-канал отдаёт ping. DoD: `POST /api/scan` кладёт job, воркер логирует его, страница `/report/[id]` показывает статус из БД.

**Фаза 2 — минимальный скан.** Краулер (ПС-04) + SSRF-защита (ПС-08) + 6 чеков: `https_ssl`, `hosting_geo`, `privacy_policy_exists`, `forms_consent_checkbox`, `cookie_banner`, `trackers_before_consent`. DoD: скан реального сайта завершается < 120 с, результаты в БД, прогресс виден через SSE.

**Фаза 3 — полный движок.** Остальные 15 чеков, скоринг, страница отчёта со светофором и блюром платной части. DoD: все 21 чек дают корректные статусы на тестовом наборе фикстур (минимум 2 фикстуры на чек: pass и fail); отдельно проверены анти-false-positive кейсы C5, E1, E2 из ПС-03.

**Фаза 4 — деньги.** ЮKassa, webhook с идемпотентностью, разблокировка отчёта, PDF через Playwright, email-доставка. DoD: тестовый платёж в песочнице ЮKassa открывает отчёт и присылает письмо с PDF.

**Фаза 5 — прод.** Юрконтур (ПС-09), антиабьюз, деплой на VPS (ПС-10). DoD: собственный домен сервиса набирает 100/100 в собственном же скане; нагрузочный тест: 10 параллельных сканов не роняют воркер.
